# Bundle Builder

A data-driven, multi-step "security system" bundle builder (React + Vite + Tailwind CSS v4): a 4-step accordion product picker on the left with a live-syncing review/checkout panel on the right, backed by Supabase with an automatic fallback to a bundled JSON catalog.

## Run it

```bash
npm install
npm run dev
```

Then open the printed local URL (default `http://localhost:5173`). `npm run build` produces a production build; `npm run preview` serves it; `npm run lint` runs ESLint.

## Backend (bonus): Supabase, with local-JSON fallback

The catalog can be served from a real Supabase project (`src/hooks/useCatalog.js`) — steps, products, variants, and shipping options each live in their own normalized table (see `supabase/seed.sql`). If Supabase is unreachable or misconfigured for any reason (missing table, network error, empty response), the app automatically falls back to the bundled `src/data/products.json` with no broken/blank state — verified by actually breaking the connection and confirming the fallback fires.

To point this at your own Supabase project:
1. Create a project, then set `VITE_SUPABASE_URL` and `VITE_SUPABASE_PUBLISHABLE_KEY` in `.env` (see `.env` for the shape).
2. Run the full contents of `supabase/seed.sql` in the Supabase SQL editor — it creates `steps`, `products`, `variants`, `shipping_options`, and `app_settings`, enables public-read RLS policies, and seeds everything from the current `products.json`.
3. `supabase/seed.sql` is generated by `node scripts/generate-seed.mjs` — re-run that any time `products.json` changes to regenerate the seed file (it's a `truncate` + reinsert, safe to re-run).
4. Product images are expected in a public Storage bucket named `product-images`; `image` fields are plain paths/URLs either way, so local dev works even without Storage set up (falls back to a placeholder tile per product on any 404).

`.env` **is committed to this repo** (intentionally, not an oversight) so the app works against the live Supabase project on a plain clone with no extra setup. This is safe here: `VITE_SUPABASE_PUBLISHABLE_KEY` is Supabase's public/anon key, meant to be exposed client-side (it ships inside any deployed app's JS bundle regardless), and every table it can reach is locked to public **read-only** via RLS. If those credentials ever stop working, the app falls back to `src/data/products.json` automatically — nothing else changes.

## Data model

Steps, products (each with a `category` and one or more `variants`), and shipping options are all data, not hardcoded per-product markup. A few flags worth knowing about, since they drive real behavior rather than just styling:
- `discount` (percent) — the "price after discount" and struck-through original price are both *derived* from `price` + `discount`, never stored as two separate numbers that could drift apart.
- `singleSelect` — marks a product (currently the plan) as pick-one/pick-none instead of a quantity; both the builder card and the review-panel line read this same flag, so they can't disagree with each other.
- `transparentImage` / `highlightTitle` — per-product display flags (skip the white image backdrop; two-tone "Cam **Unlimited**"-style title) rather than special-casing specific product IDs in components.
- `stock` (per variant, nullable = unlimited) — quantity steppers disable their `+` button at the limit.
- `shippingOptions` is an array; with one entry it renders as a plain line, with more than one it automatically becomes a selectable list — no code change needed to go from one shipping method to several.

## Notable decisions / tradeoffs

- **State**: a single `useBundle` hook (`src/hooks/useBundle.js`) owns `quantities` (per variant id), `activeVariant` (which variant's stepper a card currently shows), `selectedShippingId`, and the open accordion step. Card steppers and review-panel steppers both call the same `setQty`, so they're always in sync by construction rather than being reconciled after the fact.
- **Per-variant quantities**: each variant has its own entry in `quantities`; switching a card's color chip only changes which variant is "active" for that card's stepper — it doesn't touch other variants' counts. The review panel lists every variant across every product with qty > 0, independent of which one is currently active on its card.
- **Pricing**: totals are computed as `Σ priceAfterDiscount×qty` (and `Σ price×qty` for the struck-through total) across every line with qty > 0, plus the selected shipping option — see `src/utils/pricing.js`.
- **Persistence**: "Save my system for later" writes `{quantities, activeVariant, selectedShippingId}` to `localStorage`. On load, a saved snapshot is merged over the JSON/Supabase seed defaults; otherwise the catalog's `defaultQty` values seed the initial state. Checking out clears the saved snapshot and resets the bundle to its default state, with a confirmation dialog beforehand and a toast + button-state change afterward.
- **Loading/empty states**: a skeleton (`BundleSkeleton.jsx`) shows while the catalog loads; the review panel has a distinct empty state when nothing's selected, and each builder step handles the (currently unreachable, but defensive) case of zero available products.
- **Not finished / known gaps**:
  - No automated tests; verification was manual (dev server + build) plus scripted checks of the pricing/selection-count math and the live Supabase fallback behavior.
  - `financing.months` and `guarantee` are configurable via `app_settings` but there's no UI to edit them — they're seeded once and read-only from the app's perspective.
  - Checkout is a confirmation dialog + placeholder success state, no real order flow, per the spec ("a placeholder or a simple confirmation is fine").
